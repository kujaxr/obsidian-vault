## 核心理念

**传统 RAG vs LLM Wiki**

| 传统 RAG | LLM Wiki |
|---------|---------|
| 每次查询重新检索片段 | 增量构建持久化 wiki |
| 无知识积累 | 知识编译一次，持续更新 |
| 跨文档合成需重新推导 | 交叉引用已存在，矛盾已标记 |

> 关键区别：**wiki 是一个持久的、复合的工件。**

## 三层架构

```
┌─────────────────┐
│   Schema        │  ← CLAUDE.md / AGENTS.md (配置规范)
├─────────────────┤
│   The Wiki      │  ← LLM 生成的 markdown 文件
│   (知识库)       │    摘要、实体页、概念页、合成页
├─────────────────┤
│   Raw Sources   │  ← 原始文档 (只读)
│   (原始资料)     │    文章、论文、图片、数据文件
└─────────────────┘
```

---

## 核心操作

### 1. Ingest (摄入)
- 将新源文件放入 raw collection
- LLM 读取 → 讨论要点 → 写入 wiki 摘要页
- 更新索引、实体页、概念页
- **一个源可能触及 10-15 个 wiki 页**

### 2. Query (查询)
- 针对 wiki 提问
- LLM 搜索相关页面 → 合成答案
- **好答案可以归档回 wiki** (探索也复合)

### 3. Lint (健康检查)
- 定期检查：
  - 页面间矛盾
  - 过时声明
  - 孤立页面
  - 缺失概念页
  - 缺失交叉引用

## 特殊文件

| 文件 | 用途 |
|-----|------|
| `index.md` | 内容导向的目录，每页链接+摘要 |
| `log.md` | 时间线记录，append-only |

**log.md 技巧：**
```markdown
## [2026-04-02] ingest | Article Title
## [2026-04-03] query | Question
```
`grep "^## \[" log.md | tail -5` 查看最新 5 条

## 图像处理

**设置 Obsidian：**
1. Settings → Files → Attachment folder path: `raw/assets/`
2. Hotkeys → "Download attachments for current file" → Ctrl+Shift+D

**LLM 读取图片：** 先读文本，再单独查看引用图片

## 为什么有效

> 维护知识库的繁琐部分不是阅读或思考，而是**簿记工作**：更新交叉引用、保持摘要最新、标记矛盾、维护一致性。
> 
> 人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会无聊、不会忘记、可以一次触及 15 个文件。
> 
> **wiki 保持维护，因为维护成本接近于零。**

## 历史渊源

> 这个想法在精神上与 Vannevar Bush 的 **Memex (1945)** 相关 —— 一个个人的、策划的知识存储，文档之间有联想路径。
> 
> Bush 的愿景更接近这个，而不是后来的 web：私有、主动策划、文档间连接与文档本身一样有价值。
> 
> 他无法解决的部分是**谁来维护**。LLM 解决了这个问题。

## 使用建议

> 本文档故意抽象。它描述的是**想法**，不是具体实现。
> 
> 目录结构、schema 规范、页面格式、工具 —— 都取决于你的领域、偏好和 LLM 选择。
> 
> 正确用法：**与你的 LLM agent 分享本文档，一起实例化适合你需求的版本。**

## 应用场景

- **个人**：追踪目标、健康、心理、自我提升
- **研究**：深度主题，读论文，构建综合 wiki
- **读书**：每章归档，构建角色、主题、情节连接
- **商业/团队**：内部 wiki，由 Slack、会议、文档维护
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深度挖掘**
